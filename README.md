# PM Spec Skill Pack — 产品经理规格文档工作流

面向产品经理的规格文档工作流技能集，共 **5 个技能**，覆盖从页面文档编写、质量检查到总说明生成的完整闭环。

## 安装

### 方式一：命令行安装

```bash
# 安装全部 5 个技能，到所有已安装的 AI 编辑器（推荐）
npx skills add zhineng8888/pm-spec-skills -a '*'

# 或一行搞定：所有技能 + 所有编辑器 + 跳过确认
npx skills add zhineng8888/pm-spec-skills --all

# 指定某个编辑器
npx skills add zhineng8888/pm-spec-skills -a qoder      # Qoder
npx skills add zhineng8888/pm-spec-skills -a claude       # Claude Code
npx skills add zhineng8888/pm-spec-skills -a cursor        # Cursor

# 只安装某几个技能
npx skills add zhineng8888/pm-spec-skills --skill pm-spec-page --skill pm-spec-page-check -a '*'

# 全局安装（所有项目可用）
npx skills add zhineng8888/pm-spec-skills -g -a '*'
```

### 方式二：手动安装（任意 AI 编辑器）

技能文件就是普通的 Markdown，**任何支持 AI 指令/技能的编辑器都能使用**（Workbuddy、Trae、Cline 等）：

1. 克隆或下载本仓库
2. 将技能文件夹复制到你项目的 AI 技能目录中：
   ```
   pm-spec-doc-mode/SKILL.md      ← 悬浮球文档模式
   pm-spec-doc-mode/scripts/doc-mode.js  ← 配套 JS 引擎
   pm-spec-page/SKILL.md           ← 页面说明规范
   pm-spec-page-check/SKILL.md     ← 页面检查
   pm-spec-summary/SKILL.md        ← 总说明生成
   pm-spec-summary-check/SKILL.md  ← 总说明检查
   ```
3. 不同编辑器的技能目录不同，常见的有：
   - `.qoder/skills/`、`.claude/skills/`、`.cursor/skills/`
   - 或直接在项目根目录放一个 `Skills/` 文件夹
   - 具体路径请参考你所用编辑器的文档

## 技能列表

| 技能 | name | 用途 | 触发词 |
|------|------|------|--------|
| 悬浮球文档模式 | `pm-spec-doc-mode` | 给 HTML 原型页面注入文档模式系统（悬浮按钮 + 标注 + 页面说明） | 「给页面加文档模式」「注入悬浮球」 |
| 页面说明技能 | `pm-spec-page` | 编写页面 MD 规格文档的标准规范（章节结构、标注引用、格式约束） | 编写页面说明时自动触发 |
| 页面说明检查技能 | `pm-spec-page-check` | 对页面说明 MD 进行 10 维度质量检查（含 HTML-MD 联动检查） | 「页面检查」「检查页面文档」 |
| 总说明技能 | `pm-spec-summary` | 从各页面文档提取跨页面共性信息，生成产品总说明 | 「生成产品总说明」「串一下文档」 |
| 总说明检查技能 | `pm-spec-summary-check` | 对总说明进行 8 维度交叉验证检查 | 「检查总说明」「检查总纲」 |

## 技能依赖关系

```
pm-spec-page（页面说明编写规范）
    ├── pm-spec-page-check（页面检查，依赖页面说明规范）
    ├── pm-spec-summary（总说明生成，依赖各页面文档）
    └── pm-spec-summary-check（总说明检查，依赖页面说明规范 + 各页面文档）

pm-spec-doc-mode（悬浮球文档模式，独立技能，无依赖）
```

## pm-spec-doc-mode 引擎文件

`pm-spec-doc-mode/scripts/doc-mode.js` 是文档模式的完整 JS 引擎代码。部署时：

1. 将 `scripts/doc-mode.js` 复制到目标项目的 `shared/doc-mode.js`
2. 在目标 HTML 的 `</body>` 前注入引用：
   ```html
   <script src="../shared/doc-mode.js?v=1"></script>
   ```
3. 引擎文件无需修改，所有页面直接引用即可

## 使用指南

### 整体工作流

```
① 部署文档模式 → ② 打开文档模式 → ③ 创建标注 → ④ 编写页面说明 → ⑤ 检查页面说明 → ⑥ 生成总说明 → ⑦ 检查总说明
```

其中 **④⑤** 是每个页面的核心循环，**⑥⑦** 是所有页面写完后做一次。

---

### ① 部署文档模式（首次）

告诉 AI：

> 「给这个页面的 HTML 加上文档模式」

AI 会自动执行两件事：
1. 把 `doc-mode.js` 引擎复制到项目的 `shared/` 目录
2. 在 HTML 的 `</body>` 前注入一行 `<script>` 引用

完成后 HTML 末尾会多出：
```html
<script src="../shared/doc-mode.js?v=1"></script>
```

---

### ② 打开文档模式

**启动本地服务**（不能直接双击打开 HTML，否则浏览器会拦截文件读取）：

```bash
cd 项目根目录
python3 -m http.server 8888
```

浏览器打开 `http://localhost:8888/目录/页面.html`，页面**右下角**会出现一个绿色的 **「📄 文档」** 悬浮按钮：

- 点击 → 进入文档模式（左侧页面预览 + 右侧文档面板）
- 再次点击 → 关闭文档模式

---

### ③ 创建标注

在文档模式下：

1. 点击右侧面板的 **「标注」** 按钮 → 进入标注编辑模式
2. **鼠标移到页面元素上** → 出现绿色虚线高亮框
3. **点击高亮元素** → 自动创建标注框（带方形编号）
4. **拖动标注框四角/四边** → 调整标注范围
5. **点击左上角编号** → 可改序号
6. 点 **「✓ 完成」** → 保存标注数据到 `页面名标注.json`

标注数据会自动保存到 HTML 同目录下的 JSON 文件，下次打开自动加载。

---

### ④ 编写页面说明

告诉 AI：

> 「帮我写这个页面的页面说明」

AI 会自动：
1. 读取页面对应的标注 JSON（如果有的话）
2. 读取 HTML 页面内容
3. 按照 **6 章标准结构**生成 `页面名.md`：

| 章节 | 内容 |
|------|------|
| 一、需求背景与目标 | 入口、核心价值 |
| 二、用户角色与适用场景 | 谁用、什么时候用 |
| 三、本次新增/修改内容 | 速览清单 |
| 四、功能详细说明 | 每个功能的详细规格 |
| 五、异常处理 | 四列表格 |
| 六、验收标准 | 可验证的条件 |

写完后在文档模式下，右侧面板会自动展示页面说明内容，标注引用 `*（标注N）*` 会渲染为**绿色标签**。

---

### ⑤ 检查页面说明

告诉 AI：

> 「页面检查」或「检查页面文档」

AI 会对文档进行 **10 维度质量检查**，包括：
- 章节间一致性、需求完整性、表述清晰度
- HTML-MD 联动一致性（对比 HTML 实际内容与文档描述）
- 标注引用完整性、字段定义、业务闭环等

检查完后会输出**分级问题清单**（P0/P1/P2），每条附带**具体修改方案**，你逐条确认后才执行修改。修改完会自动复查。

---

### ⑥ 生成总说明

所有页面说明写完后，告诉 AI：

> 「生成产品总说明」或「串一下这一期的文档」

AI 会自动：
1. 扫描该期所有页面的 `.md` 规格文档
2. 提取跨页面共性信息（全局约束、数据契约、依赖关系）
3. 生成结构化的产品总说明，写入 `第X期产品总说明.md`

---

### ⑦ 检查总说明

告诉 AI：

> 「检查总说明」或「检查总纲」

AI 会对总说明进行 **8 维度交叉验证**，核心是检查总说明与各页面 MD 之间的一致性，同样输出问题清单供你逐条确认。

---

### 常见问题

**Q：标注和页面说明是什么关系？**
标注是页面上的**视觉锚点**（告诉研发“这里改了”），页面说明是**文字规格**（告诉研发“改了什么、怎么处理”）。两者通过 `*（标注N）*` 引用关联。

**Q：必须先做标注才能写页面说明吗？**
不是。标注是可选的辅助工具。没有标注也能写页面说明，有标注会让文档更直观。

**Q：页面说明和总说明的先后顺序？**
先写所有**页面说明** → 再**生成总说明** → 最后**检查总说明**。总说明是从页面说明中提炼出来的，不能反过来。

**Q：我改了页面说明后需要重新检查吗？**
建议每次改完都说「页面检查」跑一轮，确保改动没有引入新的矛盾。
