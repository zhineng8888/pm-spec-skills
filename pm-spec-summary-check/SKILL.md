# 总说明检查技能

本技能定义对总说明/总纲类 MD 文档进行质量检查的标准流程。与「页面说明检查技能」（面向单个页面MD的10维度检查）不同，本技能面向**跨页面汇总文档**，核心能力是交叉验证总说明与各页面MD之间的一致性，同时检查总说明自身的内容质量。

## 适用场景

- 总说明/产品总说明文档编写/修改完成后，提交前的自检
- 所有关联的页面MD文档完成迭代后，验证总说明是否需要同步更新
- 不涉及任何代码实现检查

## 触发指令

支持以下任一触发词：

- `检查总说明`
- `检查总纲`

## 检查流程

### Step 1：加载文档与配套文件

读取目标总说明MD文档，同时加载：
1. **pm-spec-page/SKILL.md**（页面说明规范）：确认各页面MD的章节结构规范
2. **所有关联页面MD文档**：根据总说明「页面清单」章节中列出的每个页面，逐一加载对应的页面MD文档
3. **标注JSON文件**（按需）：如果需要验证标注相关内容

### Step 2：8维度检查扫描

按以下8个维度逐项扫描，命中问题即记录并按严重程度分级：

- **P0（必须修复）**：总说明与页面MD存在严重矛盾，会导致开发理解错误
- **P1（建议修复）**：会产生歧义或信息过时，但不阻塞开发启动
- **P2（可选修复）**：轻微表述不精确，不影响核心功能

### Step 3：输出检查清单

按维度分组输出问题清单，每个问题包含：
1. 问题编号与级别（如【P1-1】）
2. 具体矛盾/缺失/模糊的描述
3. 引用原文片段（总说明 + 对应页面MD）
4. **具体修改方案**：基于文档上下文，给出可直接采用的修改文案

**修改方案要求**（与页面说明检查技能一致）：
- 必须结合文档已有的业务逻辑、术语和风格，给出**具体的修改后文字**
- 如果是缺失项，根据上下文拟写补充内容
- 如果是矛盾项，指出以哪边为准并给出统一后的文本
- 如果是过时项，指出删除或替换为当前内容
- 方案用代码块包裹，方便用户直接确认采纳

格式示例：
```
### 维度一：页面清单一致性

**【P1-1】页面清单第N行「说明」列 vs 页面MD第三章 — 描述不一致**
- 总说明：「原文A」
- 页面MD三.1-N：「原文B」
- 问题：总说明的概括与页面MD实际内容不匹配
- 修改方案：
  ```markdown
  # 页面清单第N行说明列改为：
  具体的修改后文字
  ```
- 你的决定：？
```

每个问题末尾标注「你的决定：？」，等待用户逐条回复。

### Step 4：用户确认与修复

用户逐条回复每个问题的处理方式（修复 / 不修 / 补充说明），根据回复统一修改文档。

### Step 5：复查验证

修改完成后，重新执行8维度全扫，确认所有问题已解决，输出最终检查报告。

## 8个检查维度

### 维度一：页面清单一致性

> 总说明核心：页面清单是所有页面的索引入口，必须与各页面MD保持同步

| 检查项 | 说明 |
|--------|------|
| 「是否为新页面」一致 | 总说明页面清单中的值 vs 对应页面MD头部字段，逐个对比 |
| 「说明」列准确 | 总说明对每个页面的1行概括 vs 对应页面MD第三章（本次修改/新增内容）的主要内容 |
| 「批次」与开发顺序一致 | 页面清单中的批次编号 vs 开发顺序章节中的批次划分 |
| 页面数量完整 | 总说明页面清单的行数 vs 实际存在的页面MD文件数量 |

**常见矛盾模式**：
- 页面MD头部「是否为新页面」从「否」改为「是」，但总说明仍写「否」
- 页面MD第三章新增了重要功能点，但总说明「说明」列未更新
- 某页面已从第一期移到第二期，但总说明页面清单未调整

### 维度二：全局规则交叉验证

> 总说明核心：全局规则是跨页面共享的业务约束，必须在各页面MD中有对应且一致的实现

| 检查项 | 说明 |
|--------|------|
| 数值一致 | 总说明中的次数、频率、限制等具体数值 vs 页面MD中对应功能的描述 |
| 条件一致 | 总说明中的触发条件、前置条件 vs 页面MD中的处理逻辑 |
| 覆盖完整 | 总说明列出的全局规则，在对应页面MD中都有功能描述覆盖 |
| 无孤立规则 | 总说明中的规则不是某页面MD独有逻辑的误提（应归入页面MD，不归总说明） |

**检查方法**：
1. 逐条读取总说明的全局规则
2. 在对应页面MD中搜索关键词，找到对应功能描述
3. 对比数值、条件、行为是否一致
4. 标记只属于单个页面的规则（不应出现在总说明中）

### 维度三：业务场景完整性

> 总说明核心：业务场景枚举是配置的全局字典，必须与配置页面MD中的完整列表一致

| 检查项 | 说明 |
|--------|------|
| 适用场景枚举一致 | 总说明列出的适用场景完整列表 vs 配置页面MD的字段定义表 |
| 筛选条件枚举一致 | 总说明列出的筛选条件枚举 vs 配置页面MD的联动表 |
| 新增/已有标记一致 | 总说明中标记的「本期新增」「已有」vs 页面MD中的标记 |
| 业务场景表完整 | 总说明的业务场景表格（调用方/输出形式/限制）vs 各调用方页面MD的功能描述 |
| 生成限制数值一致 | 总说明的操作限制 vs 对应页面MD中的频控规则 |

### 维度四：开发注意事项验证

> 总说明核心：开发注意事项是跨页面的关键提醒，每条必须在对应页面MD中有出处

| 检查项 | 说明 |
|--------|------|
| 有出处 | 每条注意事项能在至少一个页面MD中找到对应描述 |
| 字段存在 | 注意事项中提到的字段名/概念在对应页面MD中确实存在 |
| 表述一致 | 注意事项中的描述 vs 页面MD中的实际描述不矛盾 |
| 无过时项 | 注意事项中不包含已从页面MD中删除的功能或字段 |

**常见矛盾模式**：
- 注意事项提到某字段（如「配置编码」），但所有页面MD中已无此字段
- 注意事项描述的行为与页面MD中的最新描述不一致
- 某条注意事项对应的是历史迭代内容，当前版本已不适用

### 维度五：表述清晰度

> 与页面说明检查技能共享同一套检查规则

| 类型 | 禁用词示例 | 替代方案 |
|------|-----------|----------|
| 模糊词 | 尽快、适当、相关、大概、可能 | 给出具体数值或条件 |
| 歧义词 | 等、包括但不限于、视情况而定 | 列出所有具体项 |
| 主观词 | 小升级、大改、优化提升、通常、参考 | 用「增量变更」或列出变更项；「通常」改为明确条件；「参考」改为「必须」或「建议」 |
| 不可量化 | 体验良好、功能完善、响应快 | 给出可验证的具体标准 |
| 占位词 | 相应的、合适的、必要时 | 明确具体内容 |

**检查规则**：
- 逐一扫描全文，命中即标记
- 「等」字需判断：已列举完毕则多余，确有不全则保留
- 主观形容词一律标记

### 维度六：术语一致性

> 总说明是跨页面文档，术语一致性尤为重要——同一概念在总说明和各页面MD中必须用同一个词

| 检查项 | 说明 |
|--------|------|
| 核心术语统一 | 同一概念全文统一用词，不能同一字段有多种称呼 |
| 新旧名称不混用 | 已更名的字段（如「筛选方式」→「筛选条件」）在总说明中统一用新名称 |
| 枚举值一致 | 总说明中列出的枚举值与各页面MD中的枚举值完全一致（顺序、数量、用词） |

**检查方法**：
1. 从总说明中提取所有加粗关键词和枚举值
2. 在各页面MD中搜索同一概念
3. 检查是否使用了不同名称
4. 特别关注：旧名称残留（如总说明仍用「筛选方式」而不是「筛选条件」）

### 维度七：数据与字段定义完整性（自检）

> 总说明中涉及的字段、参数、枚举必须自身定义完整，不需要去页面MD找定义

| 检查项 | 说明 |
|--------|------|
| 字段类型明确 | 总说明中的参数表、字段表是否标注了数据类型 |
| 多字段不挤占 | 多个独立字段不能合并写在同一行或同一单元格中 |
| 枚举完整 | 总说明中列出的枚举值（如筛选条件、适用场景）是否完整，有无遗漏 |
| 数值有单位 | 所有数值是否有明确单位（如次数、秒、天、位） |
| 敏感数据处理 | 涉及用户隐私数据时，是否说明了处理方式 |

### 维度八：业务闭环与角色权限（自检）

> 总说明中的全局规则、业务场景表、AI调用链路等必须有完整的业务闭环

| 检查项 | 说明 |
|--------|------|
| 全局规则闭环 | 每条全局规则是否有完整的触发条件→处理逻辑→结果输出，无悬空规则 |
| 业务场景表完整 | 业务场景表的每个场景是否有完整的调用方、输出形式、生成限制 |
| AI调用链路闭环 | AI调用链路是否覆盖了全路径（前端→后端→AI→缓存→前端），无断点 |
| 角色明确 | 总说明中提到的每个操作是否明确了执行角色（用户/运营/系统/研发/管理员等） |
| 跨页面规则有归属 | 跨页面的规则是否在总说明中明确了由哪个页面负责实现 |
| 降级处理 | 总说明中的依赖关系（如外部配置驱动）是否说明了不可用时的降级方案 |

## 输出格式

### 检查清单格式（Step 3）

```markdown
## 总说明检查清单

### 维度N：维度名称

**【PN-N】问题标题**
- 总说明：「...」
- 页面MD：「...」（文件名 + 章节位置）
- 问题：具体矛盾/缺失/模糊描述
- 修改方案：
  ```markdown
  具体的修改后文字
  ```
- 你的决定：？

### 维度N：维度名称（通过）
— 无问题
```

| 开发注意事项验证 | ✅/❌ | 简要说明 |

## 注意事项

- 检查过程中**不修改文档**，仅输出问题清单
- 所有修复需用户逐条确认后再执行
- 修复后必须执行 Step 5 复查，确认闭环
- 交叉验证时以**页面MD为真实来源**（页面MD是更细粒度的规格文档，总说明应与之一致）
- 如果总说明和页面MD都可能有误，标记问题后由用户决定以哪边为准

## 核心约束

1. 本技能主要检查**总说明与各页面MD的一致性**（维度一~六），同时检查**总说明自身的内容质量**（维度七~八）
2. 页面MD本身的质量问题由「页面说明检查技能」负责，本技能不重复检查
3. 每个问题必须给出**具体可落地的修改建议**，禁止“建议明确”、“请补充”等空话
4. 检查清单末尾必须等待用户回复后再修改，不主动修改文档
5. 复查通过后输出最终报告，形成完整闭环
6. 交叉验证中，**页面MD是真实来源**——如果总说明与页面MD矛盾，以页面MD为准
