# 生成产品总说明

本技能用于从产品经理已编写的各页面规格文档中，自动提取跨页面共性信息，生成一份产品总说明。产品总说明的作用是：让开发团队拿到一份文档就能理解全局约束、接口契约、开发顺序，而不需要逐个翻阅每个页面的规格文档。

## 设计参考

本技能从 GitHub Spec Kit（spec-kit）中借鉴了两个具体机制，并适配产品经理视角：

1. **constitution（项目宪法）→ 全局约束**：spec-kit 用 constitution.md 定义项目不可违背的原则（编号化、可验证）。本技能将这一理念用于从各页面文档中归纳「全局约束 G-1, G-2...」，确保跨页面一致性。
2. **clarify（歧义扫描）→ 矛盾检测**：spec-kit 的 clarify 命令用 10 个分类维度（功能范围、数据模型、交互流程、边界情况、术语一致性等）结构化扫描文档歧义。本技能在步骤 3 中用类似方法检测不同页面文档间的矛盾规则，标记冲突并提示用户决策。

spec-kit 的 plan/tasks 层（技术方案、任务拆解、Phase 管理）属于开发团队职责，不在本技能范围内。

## 使用场景

1. 产品经理完成某一期所有页面文档后，需要生成产品总说明交付给开发团队
2. 新一期迭代开始时，快速生成该期的产品总说明

## 触发方式

（1）用户说「帮我生成第 X 期产品总说明」
（2）用户说「串一下这一期的文档」
（3）用户说「生成交付文档」

## 前置条件

1. 该期涉及的各页面已有规格文档（.md 格式），存放在项目对应目录下
2. index.html 中已标记各页面所属期次（按迭代批次分区）
3. 项目根目录已有「第X期产品总说明.md」文件（本技能将在该文件中追加或更新产品总说明章节）

## 核心执行步骤

### 步骤1：识别目标期次与页面范围

1. 读取 index.html，根据 HTML 中的期次分区（按迭代批次），识别目标期次包含的所有页面
2. 确认每个页面是否有对应的 .md 规格文档，列出「已有文档」和「缺失文档」
3. 若有页面缺失文档，提示用户补充后再执行，或询问是否跳过缺失页面先生成

### 步骤2：扫描所有页面文档，提取结构化信息

对每个页面的 .md 文档，提取以下信息：

| 提取项 | 来源章节 | 用途 |
|--------|----------|------|
| 页面名称、所属端 | 头部字段 | 页面清单 |
| 是否为新页面 | 头部字段 | 开发工作量评估 |
| 业务规则 | 第三章 | 提取全局共享规则 |
| 功能边界（包含/不包含） | 第四章 | 识别跨页面依赖 |
| 功能规格中的接口调用 | 第五章 | 提取数据契约 |
| 依赖条件 | 功能边界或业务规则 | 识别开发顺序 |
| 异常处理 | 异常处理章节 | 提取全局降级策略 |

### 步骤3：分析与归纳

1. **提取全局约束**：找出在多个页面文档中重复出现的规则（如统一的调用链路、频次限制等），合并为全局约束
2. **提取数据契约**：找出跨页面共享的接口格式、请求/响应结构、枚举值定义，合并为统一的数据契约
3. **分析页面依赖关系**：根据各页面「功能边界」中的「依赖条件」和接口调用关系，画出依赖图
4. **推导开发顺序**：根据依赖关系，将页面分为多个开发批次（基础设施层 → 业务页面层 → 记录/统计层）
5. **汇总注意事项**：从各页面文档中提取容易踩坑的点，统一列出

### 步骤4：生成产品总说明

按以下结构生成产品总说明内容：

```
# 第X期产品总说明

本期开发交付统一入口。各页面详细规格见对应 .md 文档。

## 一、产品定位与本期目标
（1）本期聚焦什么、不做什么
（2）核心价值主张

## 二、全局技术约束
（1）编号化的全局规则，所有页面必须遵守
（2）从各页面文档中归纳的共性约束

## 三、全局业务规则
（1）跨页面共享的业务规则（如操作次数限制、频控策略等）

## 四、数据契约
（1）跨页面共享的接口约定（请求/响应格式）
（2）枚举值清单（筛选条件、适用场景等）
（3）操作日志等通用记录格式

## 五、页面依赖关系与开发顺序
（1）依赖关系图（文字或 ASCII 图）
（2）开发批次划分（批次1/2/3...）及各批次包含的页面
（3）可并行开发的页面标注

## 六、本期涉及页面清单
表格：序号、页面名、所属端、是否新页面、开发批次、说明

## 七、外部系统接入说明（如有）
（1）外部系统的接入方式和分工

## 八、开发注意事项
（1）从各页面文档中提取的易踩坑点
（2）全局性的安全和性能要求
```

### 步骤5：写入文档

1. 将生成的产品总说明内容写入「第X期产品总说明.md」文件
2. 如果该文件已有「产品总说明」章节，则更新该章节；如果没有，则在文件末尾追加
3. 保留文件中原有的产品说明内容，不覆盖、不删除

### 步骤6：输出确认

1. 向用户展示生成结果摘要
2. 列出各页面文档的引用情况（哪些被引用了、提取了哪些关键信息）
3. 询问是否需要调整

## 核心约束规则

1. 产品总说明只提取跨页面的共性信息，不重复各页面文档中已有的细节
2. 全局约束和数据契约必须可验证、可量化，禁止模糊表述
3. 开发顺序的推导必须有明确的依赖依据，不能凭直觉排序
4. 保留原始页面文档的权威性——总纲是索引和摘要，不是替代品
5. 如果不同页面文档中出现矛盾的规则，必须标记冲突并提示用户决策
6. 文档格式规范：不使用 `>` 引用块，不使用 `-` 作为列表标记；列表统一使用 `1. 2. 3.` 或 `（1）（2）（3）` 编号

## 异常处理规则

1. 若目标期次在 index.html 中无法识别，提示用户确认页面范围
2. 若某页面文档格式不符合预期的规格文档结构，跳过该页面并记录，继续处理其他页面
3. 若提取不到任何跨页面共性信息，说明各页面完全独立，告知用户总纲可简化为页面清单 + 开发顺序
