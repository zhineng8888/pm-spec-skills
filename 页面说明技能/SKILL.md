# 页面说明文档编写规范

本技能定义产品经理为每个页面编写 MD 规格文档时的标准格式和内容要求。所有页面说明文档必须遵守此规范，确保文档一致性和可交付性。

## 文件命名与存放

- 页面说明 MD 必须与对应 HTML **同目录、同名不同后缀**
- 标注数据 JSON：`页面名标注.json`（同目录，自动生成）
- 标注说明 MD：`页面名标注.md`（同目录，可选）
- HTML 注入脚本：`</body>` 前加 `<script src="../shared/doc-mode.js?v=N"></script>`

```
模块目录/
├── 页面名.html              # 页面
├── 页面名.md                # 页面说明（本文规范约束的对象）
├── 页面名标注.json          # 标注数据（自动生成）
└── 页面名标注.md            # 标注说明（可选）
```

## MD 文档格式规范

### 头部字段（必填）

```markdown
# 页面名页面说明

所属端：xxx

所属页面：xxx.html

是否为新页面：是 / 否
```

- 三个字段分三行，中间空行
- **是否为新页面**：标识新建页面还是已有页面改造，研发据此判断影响范围
- **此字段决定后续章节结构**（见下方两种模式）

### 修改记录（正式版后新增）

头部字段之后、第一章之前，放置修改记录表。记录正式版发布之后的所有文档改动。

```markdown
## 修改记录

| 版本 | 修改时间 | 修改内容 |
|------|----------|----------|
| 1.1 | 2026-06-16 | ①xxx改动内容 |
| | | ②xxx改动内容 |
| | | ③xxx改动内容 |
| 1.0 | 2026-06-15 | 正式版本 |
```

**版本号规则**：
- **1.0**：正式版本基线，记录终审发布时间
- **1.1、1.2...**：正式版之后的每次改动，版本号递增

**格式规则**：
- 按版本号**倒序排列**（最新在上）
- 同一版本的多个改动，每个 ①② 占独立一行，版本号和日期只在第一行填写，后续行留空
- 每个 ①② 条目简要概括本次改了什么，1-2 行即可

**什么时候写**：
- 只有用户明确说「加修改记录」时才写，不主动新增
- 正式版（1.0）发布后，后续每次修改文档时，新增一行记录
- 修改前已有的条目不动，只追加新行

### 章节结构

两种模式共享同一套 6 章结构，仅第三章名称不同：

#### 模式A：新页面（是否为新页面：是）

全新页面，第三章为「**本次新增内容**」。

| 章节 | 是否必填 | 要点 |
|------|----------|------|
| 一、需求背景与目标 | 必填 | 入口、核心价值、解决什么问题 |
| 二、用户角色与适用场景 | 必填 | 谁用、什么时候用 |
| 三、本次新增内容 | 必填 | 本页面新增的功能点速览清单 |
| 四、功能详细说明 | 必填 | 每个功能的详细规格 |
| 五、异常处理 | 必填 | 四列表格：场景/触发条件/处理方式/用户提示 |
| 六、验收标准 | 必填 | 可验证的验收条件 |

#### 模式B：迭代页面（是否为新页面：否）

已有页面的迭代修改，第三章为「**本次修改内容**」。

| 章节 | 是否必填 | 要点 |
|------|----------|------|
| 一、需求背景与目标 | 必填 | 原有功能简述 + 本次迭代目标 |
| 二、用户角色与适用场景 | 必填 | 谁用、什么时候用 |
| 三、本次修改内容 | 必填 | 本次变更的速览清单 |
| 四、功能详细说明 | 必填 | 每个功能的具体变更 |
| 五、异常处理 | 必填 | 四列表格：场景/触发条件/处理方式/用户提示 |
| 六、验收标准 | 必填 | 本次变更的可验证条件 |

**统一说明**：
- 第三章是速览区，让研发快速了解本次做了什么（新增或修改）
- 不设独立的「业务规则」和「功能边界」章节——规则融入功能描述，边界通过第三章的范围自然界定
- 如有技术方案、接口设计等补充信息，可加「七、其他说明」（可选）

**第三章与第四章的关系**：
- **第三章 = 速览**：列出本次涉及的所有功能点，每个条目 1-2 行概括，**不展开**处理逻辑、异常处理等细节
- **第四章 = 展开**：基于第三章的每个条目进行详细规格描述，一个第三章条目对应一个或多个第四章功能
- **同一内容只在第四章详细写**，第三章只保留概括性描述，避免重复阅读
- 示例：
  - 第三章：「**筛选条件**：新增『按区域筛选』『按类型筛选』选项，联动展示多选组件」
  - 第四章「功能1」：展开下拉选项、联动逻辑、组件交互、标注引用等完整规格

**第三章一致性约束**：

第三章（本次修改/新增内容）是整份文档的「错点」，驱动并约束第四、五、六、七章的全部内容。第一章和第二章仅提供背景上下文。

*正向约束（三 → 四五六七）：*
- 第三章每一条必须能在第四章找到对应的功能详细展开
- 第三章涉及的功能必须有对应的验收标准（第六章）
- 第三章涉及的功能如有异常情况，必须列入异常处理（第五章）

*反向约束（四五六七 → 三）：*
- 在四、五、六、七章新增内容时，必须检查第三章是否已包含对应条目
- 如果第四章新增了功能，但第三章没有对应条目，必须回补到第三章
- 这个反向检查保证第三章始终是文档的完整索引

*检查流程：*
1. 头部字段：「是否为新页面」与实际内容一致（弹窗改独立页面 = 新页面）
2. 第三章每一条 → 检查第四章是否有对应功能
3. 第三章每一条 → 检查第六章是否有对应验收标准
4. 第四章每个功能 → 反向检查第三章是否已列出
5. 第五章每条异常 → 对应功能是否在第三章范围内

### 格式要点

- 标题用 `# 页面名页面说明`
- 章节用中文数字（一、二、三...），子项用阿拉伯数字（1. 2. 3.）
- 子项的子项用（1）（2）（3），不用短横线「-」
- 功能标题用「功能1、功能2」不用「模块」
- ★ 为纯文本星号，不用 emoji
- 不使用 `---` 分隔线
- `**加粗**` 和 `*斜体*` 均可使用，但 `*斜体*` 会被渲染为绿色高亮标签（见渲染规则）

## 功能详细说明编写规范

### 功能标题

统一使用 `### **功能N：功能名称**`（加粗标题），如 `### **功能1：筛选条件联动多选**`。

功能标题同时使用 `###` 和 `**加粗**`，确保在文档模式中渲染为醒目的加粗标题样式，而不是普通的 h3 文字。

### 功能描述格式

根据功能复杂度选择合适格式：

**格式A：四要素格式**（复杂功能，含交互流程）

```markdown
### **功能N：功能名称**

**用户场景**：描述谁在什么情况下使用

**输入**：输入参数说明

**处理逻辑**：
1. 具体处理步骤
2. ...

**输出**：输出结果说明
```

**格式B：简化格式**（简单功能，纯逻辑或纯变更）

```markdown
### **功能N：功能名称**

*（标注N）* **变更内容**：描述变更了什么

（1）具体变更项1
（2）具体变更项2
```

或直接用加粗标签：

```markdown
### **功能N：保存**

保存校验：检查必填字段（字段1、字段2），未填则 alert 提示，全部填完正常保存。
```

**选择原则**：
- 涉及多步骤交互流程 → 用格式A（四要素）
- 纯选项新增、纯字段变更、简单校验 → 用格式B（简化）
- 不必每个功能都凑齐四要素，按实际需要选择

### 保存功能写法

保存功能的描述聚焦校验逻辑，不需要单独说明取消操作：

```markdown
### **功能N：保存**

保存校验：检查必填字段是否已填写（字段1、字段2；条件字段3）。未填则 alert 提示对应字段，全部填完则正常保存。
```

**说明**：取消编辑属于页面通用导航行为（直接返回列表页），不需要在功能描述中单独说明。

## 异常处理规范

异常处理统一采用 **四列表格**：

```markdown
| 序号 | 异常场景 | 触发条件 | 处理方式 | 用户提示 |
|------|----------|----------|----------|----------|
| 1 | 具体异常 | 触发时机 | 系统行为 | 用户看到的提示 |
```

**编写规则**：
1. 每个异常独立一行，不合并场景
2. 「用户提示」填写用户实际看到的提示文案，无提示则填「无」
3. 后端异常也需覆盖，用户提示填「无」或具体 Toast 文案
4. 触发条件要具体，如「点击保存（筛选条件为按区域或按类型）」

## 标注引用格式

标注是页面 HTML 上的绿色位置标签，通过文档模式创建，用于将 MD 文档描述与页面 UI 元素关联起来。

### 标注引用写法

```markdown
# 单个标注
*（标注1）* **筛选条件**（原「筛选方式」）：新增「按区域筛选」选项

# 多个标注组合
*（标注6、7）* 新增「记录ID」和「用户ID」两个Tab

# 表格中使用
| 3 | 适用场景*（标注3）* | 该模板适用的业务场景 |

# ** 和 * 之间必须加空格
**筛选条件新增选项** *（标注1）*：在原有选项基础上新增...
```

### 引用规则

- 格式必须是 `*（标注N）*`：星号 + 中文全角括号 + “标注” + 数字 + 中文全角括号 + 星号
- 多个标注组合：`*（标注1、3）*`，顿号分隔，不加空格
- 写在文字流中，不要单独成行
- `**` 和 `*` 相邻时**必须加空格**（`**选项** *（标注1）*`），否则渲染异常
- 标注位置标签仅作视觉引导，标注框实际位置由 JSON 中的 xpath/坐标决定

### 标注工作流程

**Step 1：读取标注数据**

编写 MD 前，先读取对应 HTML 的标注 JSON 文件（`[页面名]标注.json`），了解页面上已有哪些标注点：

```json
{
  "id": 1,
  "keyTexts": ["筛选条件", "*"],
  "xpath": "//div.../label",
  "context": "main"
}
```

- `id`：标注编号，用于 MD 中引用
- `keyTexts`：标注指向的页面元素文字，帮助理解标注指哪里
- `context`：标注所在的页面区域

**Step 2：在 MD 中引用标注**

将标注引用放在对应描述的**最前面**，让研发一眼看到 UI 位置：

```markdown
## 三、本次修改内容

1. *（标注1）* **筛选条件**（原「筛选方式」）：
   （1）*（标注2）* 新增「按区域筛选」选项：联动展示区域多选组件
   （2）*（标注3）* 新增「按类型筛选」选项：联动展示类型多选组件
2. *（标注4）* **适用场景** 新增4个选项
```

**最佳实践**：
- **第三章（速览区）**：每个修改点开头都放标注引用，让研发快速定位页面元素
- **第四章（功能详细说明）**：在处理逻辑中按需引用标注，不必每条都加
- 没有标注的功能（如保存校验、纯后端逻辑）不加强行加标注

**Step 3：创建新标注**

如果页面还没有标注，或需要新增：
1. 在页面中完成标注（文档模式 → 标注 → 完成）
2. 标注数据自动保存到 `[页面名]标注.json`
3. AI 读取标注 JSON，识别每个标注指向的元素
4. AI 在页面说明 MD 对应位置插入 `*（标注N）*` 引用
5. 保存 MD，重新打开文档模式即可看到绿色标签

## MD 渲染规则（写文档时需注意）

悬浮球文档模式内置轻量 Markdown→HTML 渲染器，**只支持以下语法**：

| 语法 | 渲染效果 | 注意事项 |
|------|----------|----------|
| `#` `##` `###` | h1/h2/h3 标题 | 仅三级 |
| `**加粗**` | 加粗文本 | |
| `*斜体*` | **绿色高亮标签** | 不是传统斜体，是绿色背景 |
| `` `代码` `` | 行内代码 | |
| ` ```代码块``` ` | 围栏代码块 | 浅灰背景+圆角，语言标识符自动隐藏 |
| `[text](url)` | 可点击链接 | 蓝色下划线 |
| `\| 表格 \|` | 表格 | 自动处理分隔行 |
| 纯 URL | 自动转链接 | |

### 不支持的语法（写文档时避免使用）

- 有序/无序列表嵌套（简单列表可以，深层嵌套可能渲染异常）
- 引用块 `>`
- 图片 `![](url)`
- HTML 标签（会被转义显示）
- 脚注、定义列表等高级语法

### 关键注意点

- `*斜体*` 渲染为**绿色高亮**（非传统斜体），用于标注引用 `*（标注N）*` 和需要强调的关键词
- 围栏代码块内容自动 HTML 转义，可以安全放置 JSON/XML 等示例
- 代码块内的 `<span>` 等标签会显示为原始文本，不会被渲染

## 核心约束

1. 页面说明 MD 是**规格文档**，不是技术文档——描述 WHAT（做什么），不描述 HOW（怎么实现）
2. 两种模式共享 6 章结构，仅第三章名称不同：新页面「本次新增内容」/ 迭代页面「本次修改内容」
3. 不设独立的「业务规则」和「功能边界」章节——规则融入功能描述，边界通过第三章自然界定
4. **第三章是速览，第四章是展开**——同一内容只在第四章详细写，第三章不重复实现细节
5. **第三章是文档锚点**——正向约束四五六七章内容覆盖，反向检查四五六七章新增内容是否已回补到第三章
6. 功能描述按复杂度选择格式：复杂交互用四要素，简单变更用简化格式
7. 异常处理用四列表格，覆盖所有可能的异常情况
8. 验收标准必须可验证，避免“体验良好”等模糊表述
9. 取消编辑等通用导航行为不需要在功能描述中单独说明
