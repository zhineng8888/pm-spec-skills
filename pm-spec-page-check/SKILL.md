# 页面说明检查技能

本技能定义对页面说明MD文档及其对应的HTML原型页面进行质量检查的标准流程。综合业务层检查和需求质量单测的核心思路，精简为适合页面说明文档的10维度检查体系，其中第7维度为HTML-MD联动检查，第8~10维度为内容质量深度检查。

## 适用场景

- 页面说明MD文档编写/修改完成后，提交前的自检
- HTML原型页面修改后，检查MD文档是否同步更新
- 文档层面的逻辑一致性、完整性、清晰度检查
- HTML原型与MD文档之间的交叉一致性检查

## 触发指令

支持以下任一触发词：

- `页面检查`
- `检查页面文档`

## 检查流程

### Step 1：加载文档与配套文件

读取目标页面说明MD文档，同时加载：
1. **pm-spec-page/SKILL.md**（页面说明规范）：确认文档章节结构是否符合规范
2. **标注JSON文件**（同目录下的`页面名标注.json`，如存在）：用于标注引用完整性校验
3. **对应HTML原型页面**（同目录下的`页面名.html`）：用于第7维度HTML-MD联动检查
4. **识别文档模式**：根据头部「是否为新页面」判断第三章名称（新页面=「本次新增内容」/ 迭代页面=「本次修改内容」）

**HTML文件定位规则**：
- 根据MD头部「所属页面」字段获取HTML文件名
- HTML文件与MD文件位于同一目录下
- 如果HTML文件不存在，第7维度跳过并输出提示

### Step 2：10维度检查扫描

按以下10个维度逐项扫描，命中问题即记录并按严重程度分级：

- **P0（必须修复）**：章节间存在严重矛盾，阻塞开发理解
- **P1（建议修复）**：会产生歧义或遗漏，但不阻塞开发启动
- **P2（可选修复）**：轻微表述不清，不影响核心功能

### Step 3：输出检查清单

按维度分组输出问题清单，每个问题包含：
1. 问题编号与级别（如【P1-1】）
2. 具体矛盾/缺失/模糊的描述
3. 引用原文片段
4. **具体修改方案**：基于文档上下文，给出可直接采用的修改文案（不是“建议明确”“请补充”等空话）

**修改方案要求**：
- 必须结合文档已有的业务逻辑、术语和风格，给出**具体的修改后文字**
- 如果是缺失项，根据上下文拟写补充内容
- 如果是矛盾项，指出以哪边为准并给出统一后的文本
- 如果是模糊表述，给出替换后的清晰表述
- 方案用代码块包裹，方便用户直接确认采纳

格式示例：
```
### 维度一：章节间一致性

**【P1-1】功能X第N条 vs 验收标准第M条 — 描述矛盾**
- 功能X：「原文A」
- 验收标准M：「原文B」
- 问题：两者口径不一致
- 修改方案：
  ```markdown
  # 验收标准第M条改为：
  M. 具体的修改后文字
  ```
- 你的决定：？
```

每个问题末尾标注「你的决定：？」，等待用户逐条回复。

### Step 4：用户确认与修复

用户逐条回复每个问题的处理方式（修复 / 不修 / 补充说明），根据回复统一修改文档。

### Step 5：复查验证

修改完成后，重新执行10维度全扫，确认所有问题已解决，输出最终检查报告。

## 10个检查维度

### 维度一：章节间一致性

> 来源：Spec Kit analyze 命令（跨文档一致性检查）

检查页面说明MD内部各章节之间对同一功能/概念的描述是否一致。两种模式共享同一套章节映射（仅第三章名称不同）：

| 交叉检查组合 | 检查内容 |
|-------------|----------|
| 三（新增/修改内容）↔ 四（功能详细说明） | 三列出的每个功能点，四是否都有对应且描述一致 |
| 四（功能详细说明）↔ 五（异常处理） | 四的每个功能是否都有异常处理条目覆盖 |
| 四（功能详细说明）↔ 六（验收标准） | 四的每个功能是否都在验收标准中有对应 |
| 三（新增/修改内容）↔ 六（验收标准） | 三提到的数量/范围/场景数是否与六一致 |

**常见矛盾模式**：
- 数量不一致（如功能描述说"4个场景"，验收标准说"7个场景"）
- 功能点遗漏（三列了5个功能，四只写了4个）
- 保存校验规则与异常处理表格不对应

### 维度二：需求完整性

> 来源：Spec Kit checklist（需求质量单测 — Completeness 维度）

| 检查项 | 说明 |
|--------|------|
| 异常覆盖 | 每个功能是否都有对应的异常处理条目 |
| 验收覆盖 | 每个功能是否都在验收标准中有对应 |
| 字段定义 | 保存校验中提到的字段是否都在功能描述中定义过 |
| 必填项声明 | 必填的多选/输入组件是否在三或四中标注了「必填」 |
| 空状态处理 | 列表类组件是否说明了数据为空时的展示 |

### 维度三：表述清晰度

> 来源：需求表述清晰度检查 + 禁用表述清单

扫描文档中是否存在以下禁用/慎用表述：

| 类型 | 禁用词示例 | 替代方案 |
|------|-----------|----------|
| 模糊词 | 尽快、适当、相关、大概、可能 | 给出具体数值或条件 |
| 歧义词 | 等、包括但不限于、视情况而定 | 列出所有具体项 |
| 主观词 | 小升级、大改、优化提升、通常、参考 | 用「增量变更」或列出变更项；「通常」改为明确条件；「参考」改为「必须」或「建议」 |
| 不可量化 | 体验良好、功能完善、响应快 | 给出可验证的具体标准 |
| 占位词 | 相应的、合适的、必要时 | 明确具体内容 |

**检查规则**：
- 逐一扫描全文，命中即标记
- 「等」字需判断：已列举完毕则多余，确有不全则保留
- 主观形容词（如「小升级」「大幅」）一律标记

### 维度四：术语一致性

> 来源：Spec Kit clarify（Terminology & Consistency 维度）

| 检查项 | 说明 |
|--------|------|
| 同一概念全文统一用词 | 如“筛选条件”不能有时叫“筛选方式” |
| 组件名称统一 | 如“区域多选组件”全文一致 |
| 功能名称统一 | 如三叫"增量变更"，四也叫"增量变更" |

**检查方法**：
1. 提取全文中所有加粗关键词
2. 检查同一概念是否在不同章节使用了不同名称
3. 特别关注：旧名称 vs 新名称（如原名 → 新名的过渡是否全文统一）

### 维度五：验收标准质量

> 来源：验收标准检查

| 检查项 | 说明 |
|--------|------|
| 可验证性 | 每条验收标准是否可量化、可测试 |
| 判定条件 | 是否有明确的成功/失败判定 |
| 覆盖度 | 是否覆盖了所有核心功能的验收 |

**禁用表述**：
- 「功能正常」→ 列出具体功能点和预期输出
- 「体验OK」→ 给出具体体验指标
- 「满足要求」→ 明确具体要求内容

### 维度六：标注引用完整性

> 来源：项目特有需求（标注系统与MD文档的关联校验）

| 检查项 | 说明 |
|--------|------|
| MD→JSON | MD中引用的每个标注编号（如标注1-11），JSON中是否都存在 |
| JSON→MD | JSON中的每个标注ID，MD中是否都有引用 |
| 格式正确性 | 标注引用格式是否为 `*（标注N）*`（星号+全角括号） |
| JSON是否存在 | 如果同目录下不存在标注JSON文件，本维度直接通过（无需标注） |

**检查方法**：
1. 从MD中正则提取所有 `标注N` 中的数字N
2. 从JSON中提取所有 `"id": N`
3. 对比两个集合，找出：MD有但JSON无、JSON有但MD无的编号
4. 如果JSON文件不存在，输出「标注JSON不存在，跳过本维度」

### 维度七：HTML-MD联动一致性

> 来源：HTML原型页面与MD文档的交叉校验

将HTML原型页面作为“实现的真相”，检查MD文档描述是否与HTML实际行为一致。

| 检查项 | 说明 |
|--------|------|
| 页面结构一致性 | MD中描述的卡片/区域/按钮是否在HTML中都存在；HTML中是否有MD未提及的元素 |
| 交互流程一致性 | MD中描述的点击行为、显隐逻辑、状态切换是否与HTML的JS逻辑一致 |
| 文案一致性 | MD中引用的Toast提示、按钮文案、标签文字是否与HTML中的实际文案完全一致 |
| 元素ID/引用一致性 | JSON标注文件中的xpath/ID是否在当前HTML中存在（如存在标注JSON） |
| 样式一致性 | MD配色规范中的色值是否与HTML CSS中的实际色值一致 |
| 数据一致性 | MD中描述的静态数据（如快捷按钮数量、文案池内容）是否与HTML中的实际数据一致 |

**检查方法**：
1. 读取HTML文件全文，提取所有可见文案、按钮文字、Toast提示、CSS色值、元素ID
2. 与MD文档逐项比对：
   - MD提到的每个按钮文案 → HTML中是否存在且文字一致
   - MD提到的每个Toast提示 → HTML中 `showToast('xxx')` 是否一致
   - MD提到的每个色值（如 `#3ecf8e`） → HTML CSS中是否使用
   - MD提到的每个元素（如“模板标签”“重新生成卡片”） → HTML中是否有对应DOM
3. HTML中存在但MD未提及的功能/元素 → 标记为文档缺失项
4. MD中描述但HTML中不存在的功能 → 标记为文档过时项

**常见矛盾模式**：
- Toast文案不一致（如MD写「已打开分享页面」但HTML写「已打开分享」，文案差异）
- 按钮文案不一致（如MD写「AI 优化文案」但HTML写「AI优化文案」，空格差异）
- 功能已在HTML中删除但MD未同步更新
- 色值变更后MD配色规范未更新

### 维度八：数据与字段定义完整性

> 来源：数据要求检查

检查文档中涉及的数据字段是否有完整的定义：

| 检查项 | 说明 |
|--------|------|
| 字段类型明确 | 每个输入/输出字段是否标注了数据类型（string/int/boolean/datetime/text） |
| 长度与格式 | 字符串字段是否有最大长度；ID字段是否说明了位数（如18位）；日期字段是否有格式（如 YYYY-MM-DD） |
| 取值范围 | 数值字段是否有最小值/最大值；枚举字段是否列出了所有可选值 |
| 多字段不挤占 | 多个独立字段不能合并写在同一行或同一单元格中（如「城市公司/城市/品牌」应拆为三个字段） |
| 敏感数据处理 | 涉及用户隐私数据（手机号、姓名、访问轨迹）时，是否说明了脱敏/加密/不留存等处理方式 |

**常见命中模式**：
- 表格中一行包含多个字段名用斜杠分隔（如「输入/输出内容」），应拆为独立行
- ID字段未说明位数（如「用户ID」未标注18位）
- 数据包中包含手机号但未说明脱敏方式
- 数值字段（如排序值、积分数量）未说明最小值/最大值

### 维度九：业务流程闭环与角色权限

> 来源：业务逻辑一致性 + 角色与权限检查

| 检查项 | 说明 |
|--------|------|
| 流程闭环 | 每个核心业务流程是否有完整的起始→处理→结束路径，无悬空步骤 |
| 异步回调闭环 | 异步任务（如AI生成、通知推送）是否有完成后的回调处理说明 |
| 通知→跳转闭环 | 涉及IM/推送通知的场景，用户点击通知后的跳转目标是否明确；跳转时如果数据已更新，展示的是哪个版本 |
| 角色明确 | 每个操作是否明确了执行角色（用户/运营/系统/管理员等） |
| 权限边界 | 不同角色能看到/操作的功能是否有清晰边界（如「不同用户共享报告」是否有完整说明） |
| 跨角色数据隔离 | 当一个数据被多个角色访问时，是否说明了各自看到的内容差异 |

**常见命中模式**：
- 描述了「发送IM通知」但未说明点击通知后的行为
- 描述了「生成报告」但未说明其他角色再次进入时看到什么
- 操作描述中未指明执行者是谁（如「点击按钮」没说谁点）
- 涉及多角色共享数据时，未说明数据更新后各角色看到的版本

### 维度十：边界条件覆盖

> 来源：验收标准 + 边界条件检查

| 检查项 | 说明 |
|--------|------|
| 空值处理 | 输入为空、数据为空、列表为空时是否有明确处理 |
| 极值处理 | 达到上限/下限时的行为是否说明（如次数用尽、排序值为0） |
| 并发/重复处理 | 同一操作被多次触发时如何处理（如重复点击生成、多人同时操作） |
| 时间边界 | 跨天/跨月场景是否有说明（如「当天生成限制」跨天后如何处理） |
| 降级处理 | 依赖服务不可用时的降级方案是否明确（如外部服务无数据时如何处理） |

**检查方法**：
1. 对每个功能的核心参数，检查是否覆盖了：空值、最小值、最大值、边界值
2. 对每个频控/限制规则，检查是否覆盖了：达到上限、跨周期重置、并发触发
3. 对每个外部依赖，检查是否覆盖了：不可用、超时、返回异常

## 输出格式

### 检查清单格式（Step 3）

```markdown
## 页面文档检查清单

### 维度N：维度名称

**【PN-N】问题标题**
- 问题描述：...
- 引用原文：「...」
- 修复建议：...
- 你的决定：？

### 维度N：维度名称（通过）
— 无问题
```

### 最终报告格式（Step 5）

```markdown
## 页面文档最终检查报告

| 维度 | 结果 | 说明 |
|------|:---:|------|
| 章节间一致性 | ✅/❌ | 简要说明 |
| 需求完整性 | ✅/❌ | 简要说明 |
| 表述清晰度 | ✅/❌ | 简要说明 |
| 术语一致性 | ✅/❌ | 简要说明 |
| 验收标准质量 | ✅/❌ | 简要说明 |
| 标注引用完整性 | ✅/❌/⚠️跳过 | 简要说明（JSON不存在时标记跳过） |
| HTML-MD联动一致性 | ✅/❌/⚠️跳过 | 简要说明（HTML不存在时标记跳过） |
| 数据与字段定义完整性 | ✅/❌ | 简要说明 |
| 业务流程闭环与角色权限 | ✅/❌ | 简要说明 |
| 边界条件覆盖 | ✅/❌ | 简要说明 |

**结论**：N个P0，N个P1，N个P2
```

## 注意事项

- 检查过程中**不修改文档**，仅输出问题清单
- 所有修复需用户逐条确认后再执行
- 修复后必须执行 Step 5 复查，确认闭环
- 异常处理表格遵循页面说明技能规范：四列表格（场景/触发条件/处理方式/用户提示）
- 标注引用格式遵循页面说明技能规范：`*（标注N）*`

## 核心约束

1. 本技能检查的是**文档质量**（写得好不好），不是**实现正确性**（做得对不对）
2. 每个问题必须给出**具体可落地的修改建议**，禁止"建议明确"、"请补充"等空话
3. 检查清单末尾必须等待用户回复后再修改，不主动修改文档
4. 复查通过后输出最终报告，形成完整闭环
